# M&D 登录后三功能主题 UX 设计稿

## 1. 设计结论

登录后的三个“主题”不是视觉主题，也不是 Feed 内的三个筛选板块，而是三套功能完整不同的 App 模式：

- 宠物交流：社区动态、圈子、发布、互动消息和个人主页。
- 宠物收养：待领养浏览、救助动态、领养申请、救助沟通和收养档案。
- 宠物交易：宠物用品和服务市集、发布交易、买卖沟通、订单和卖家资料。

Stitch 视觉主题仍然是 Honey、Mint、Night、Neutral，负责颜色、背景和组件观感。功能主题负责信息架构、底部导航、首页内容、详情页、消息上下文和设置中的默认进入模式。两者必须分开实现。

## 2. 全局进入规则

### 2.1 登录后路由

- 用户已有默认功能主题时，登录后直接进入该主题的首页。
- 用户没有默认功能主题时，登录后展示“选择你的 M&D 入口”页面。
- 选择功能主题后进入对应 App Shell，并持久化为默认功能主题。
- 设置页可以随时切换默认功能主题，切换后整套底部导航和功能入口一起变化。

### 2.2 首次选择页面

页面标题：选择你的 M&D 入口

副标题：你可以之后在设置里随时切换。

三个入口卡片：

- 宠物交流
  - 主文案：分享猫猫日常，加入同好圈子。
  - 功能预览：动态、圈子、发布、消息、我的。
  - 主按钮：进入交流。

- 宠物收养
  - 主文案：给待安家的猫猫和 doggie 一个安全入口。
  - 功能预览：领养、救助、申请、消息、我的。
  - 主按钮：进入收养。

- 宠物交易
  - 主文案：浏览用品、服务、交换和同城交易。
  - 功能预览：市集、发布、消息、订单、我的。
  - 主按钮：进入交易。

视觉要求：

- 使用 Stitch 16dp 消费卡片圆角。
- 卡片内可以用真实猫猫或 doggie 图片，不使用装饰渐变块。
- 当前视觉主题只影响颜色和背景，不改变功能卡片结构。
- 三张卡必须有相同高度、相同主按钮位置，避免移动端布局跳动。

## 3. App Shell 结构

每个功能主题都有自己的 TopBar、首页、底部导航和页面栈。

公共规则：

- TopBar 左侧保留 M&D 品牌标识。
- TopBar 中间显示当前功能主题名称。
- TopBar 右侧保留通知、搜索或设置入口，具体图标按当前功能主题决定。
- 底部导航由当前功能主题生成，不使用固定的 Feed / Market / Messages / Orders / Profile。
- 切换功能主题后默认回到新主题的首页 tab。
- 如果当前有未保存的发布内容、领养申请或交易表单，切换前需要提示保存草稿或放弃。

## 4. 三套底部导航

### 4.1 宠物交流底部导航

底部导航：

1. 动态
2. 圈子
3. 发布
4. 消息
5. 我的

页面职责：

- 动态：社区信息流，猫猫内容优先，doggie 内容作为友好分支。
- 圈子：猫猫新手村、橘猫联盟、猫咪摄影、同城遛狗等圈子入口。
- 发布：发动态、发图片、选择圈子、添加标签。
- 消息：评论、点赞、关注、私信。
- 我的：个人资料、我的宠物、收藏动态、关注圈子。

交流模式不展示订单、购物车、领养申请主流程。可以在动态中出现收养或交易推荐卡，但点击后必须明确跳转到对应功能主题。

### 4.2 宠物收养底部导航

底部导航：

1. 领养
2. 救助
3. 申请
4. 消息
5. 我的

页面职责：

- 领养：待领养宠物列表、搜索、筛选、推荐卡片和安全提示。
- 救助：救助站动态、救助故事、成功安家案例、救助人资料。
- 申请：我的领养申请列表、申请状态、补充资料入口。
- 消息：围绕领养申请和救助沟通的会话，不混入买卖订单语境。
- 我的：收养偏好、收藏的待领养宠物、申请记录、家庭资料草稿。

收养模式禁止出现：

- 购物车
- 订单
- 立即购买
- 加入购物袋
- 支付

收养模式允许出现：

- 申请领养
- 联系救助人
- 收藏
- 补充资料
- 查看申请状态

### 4.3 宠物交易底部导航

底部导航：

1. 市集
2. 发布
3. 消息
4. 订单
5. 我的

页面职责：

- 市集：用品、服务、交换、求购、免费赠送和推荐商品。
- 发布：发布出售信息、服务信息、交换信息或赠送信息。
- 消息：买家和卖家沟通、询价、议价、服务预约。
- 订单：待付款、进行中、已完成、售后。
- 我的：买家资料、卖家资料、发布管理、交易评价。

交易模式可以保留立即购买、联系卖家、出价、收藏等动作，但不允许把活体宠物收养做成交易订单。

## 5. 宠物收养详细设计

### 5.1 领养首页

页面顺序：

1. 顶部标题区
   - 标题：宠物收养
   - 副标题：给待安家的猫猫和 doggie 一个安全入口。

2. 搜索框
   - 占位：搜索城市、品种、救助站...

3. 筛选 chips
   - 全部
   - 猫咪
   - Doggie
   - 幼年
   - 已免疫
   - 已绝育
   - 同城

4. 安全提示卡
   - 文案：M&D 不支持活体买卖。领养前请确认救助方信息，了解回访、疫苗、绝育和长期照护责任。

5. 待领养卡片列表
   - 图片
   - 状态 chip：待领养、审核中、已预约、已安家
   - 宠物名
   - 基本信息：猫咪 / 3 个月 / 已驱虫 / 上海
   - 性格标签：亲人、安静、适合新手
   - 救助方：城南救助 / 已认证
   - 主按钮：申请领养
   - 次按钮：联系

6. 成功安家故事
   - 以 2 到 3 张小卡展示，不抢列表主线。

### 5.2 领养详情

页面结构：

- 顶部导航：返回、宠物名、分享、收藏。
- 图片轮播。
- 基本信息：名字、状态、年龄、性别、城市。
- 健康信息：驱虫、疫苗、绝育、体检备注。
- 性格与照护：亲人程度、活动量、是否适合新手、是否需要封窗。
- 领养要求：稳定住所、接受回访、不弃养、家庭成员同意。
- 救助方信息：头像、名称、认证状态、响应速度。
- 申请流程：提交资料、沟通、回访、试养、确认安家。
- 底部固定操作：联系救助人、申请领养。

### 5.3 领养申请

表单字段：

- 居住城市。
- 居住情况：自有、租住、合租、宿舍、其他。
- 是否封窗或有安全防护。
- 家中已有宠物。
- 养宠经验。
- 是否接受回访。
- 申请说明。
- 勾选确认：我理解 M&D 不支持活体买卖，领养需要尊重动物福利。

动作：

- 保存草稿。
- 提交申请。

申请状态：

- 草稿
- 已提交
- 沟通中
- 待回访
- 通过
- 未通过
- 已安家

## 6. 设置页设计

设置页需要把“功能主题”和“视觉主题”分成两个入口。

偏好区：

- 默认功能主题：宠物交流 / 宠物收养 / 宠物交易
- 外观与主题：Honey / Mint / Night / Neutral
- 通知设置

默认功能主题页面：

- 标题：默认功能主题
- 说明：选择登录后默认进入的 M&D 功能区。
- 三张选项卡：宠物交流、宠物收养、宠物交易。
- 每张卡展示对应底部导航预览。
- 操作按钮：应用并进入。

交互规则：

- 点击卡片只改变当前选择态。
- 点击“应用并进入”后保存默认功能主题，并切换整套 App Shell。
- 视觉主题不受影响。
- 切换后新主题首页 tab 选中。
- 如果当前有未保存内容，先弹出确认。

## 7. 消息规则

消息是每个功能主题都可能有的通用能力，但消息上下文必须跟随功能主题变化。

宠物交流消息：

- 评论
- 点赞
- 关注
- 圈子通知
- 私信

宠物收养消息：

- 领养申请上下文卡
- 救助人沟通
- 补充资料提醒
- 回访安排
- 申请状态变化

宠物交易消息：

- 商品咨询
- 议价
- 服务预约
- 订单状态
- 售后沟通

同一个“消息”tab 的视觉结构可以复用，但列表筛选、上下文卡、空状态和详情页标题必须按功能主题变化。

## 8. 个人页规则

“我的”也是跨主题通用 tab，但内容必须按功能主题变化。

宠物交流的我的：

- 个人资料
- 我的宠物
- 我的动态
- 我的圈子
- 收藏动态

宠物收养的我的：

- 家庭资料
- 收养偏好
- 收藏的待领养宠物
- 我的申请
- 救助沟通记录

宠物交易的我的：

- 买家资料
- 卖家资料
- 发布管理
- 订单
- 评价

## 9. 可访问性要求

- 底部导航每个 item 最小触控区域 48dp。
- 当前功能主题和当前 tab 不能只靠颜色表达，需要同时有文字、图标状态或选中指示。
- 功能主题选择卡可被 TalkBack 读出：名称、说明、底部导航预览、是否已选中。
- 收养宠物图片必须有 contentDescription，例如“橘子，待领养猫咪照片”。
- 表单字段必须有 label，错误提示显示在字段下方。
- 切换功能主题时，如果有未保存内容，确认弹窗需要明确说明影响。

## 10. KMP Android 实现约束

建议命名：

- 功能主题：`MndAppMode`
- 取值：`community`、`adoption`、`trade`
- 持久化 key：`mnd_app_mode`

不要命名为 `MeowTheme`，也不要和 Honey / Mint / Night / Neutral 混在一起。

导航建议：

- `CommunityNavGraph`
- `AdoptionNavGraph`
- `TradeNavGraph`

底部导航建议由当前 `MndAppMode` 派生：

- `community` -> 动态、圈子、发布、消息、我的。
- `adoption` -> 领养、救助、申请、消息、我的。
- `trade` -> 市集、发布、消息、订单、我的。

落地重点：

- 不要在 `StitchFeedScreen` 里只加三个切换按钮。
- 不要把收养做成 Market 的一个 filter。
- 不要让固定底部导航继续出现在所有模式里。
- 可以复用卡片、TopBar、空状态、消息列表组件，但页面语义和导航结构必须分开。

## 11. 第一版落地范围

第一版建议先做：

1. 登录后默认功能主题路由。
2. 无默认功能主题时的三入口选择页。
3. 三套底部导航。
4. 宠物交流模式首页和已有社区能力接入。
5. 宠物交易模式市集、发布、消息、订单、我的接入。
6. 宠物收养模式的领养、救助、申请、消息、我的。
7. 设置页默认功能主题切换。
8. 收养详情和领养申请表单。

暂不做：

- 复杂救助机构后台审核。
- 实名认证全流程。
- 支付化领养。
- 跨主题混合信息流。

## 12. 验收标准

- 登录后能进入默认功能主题。
- 没有默认功能主题时，先显示三入口选择页。
- 选择宠物交流、宠物收养、宠物交易后，底部导航确实不同。
- 设置页可以切换默认功能主题，并在应用后进入对应 App Shell。
- 宠物收养有独立的领养、救助、申请、消息、我的五个 tab。
- 宠物收养不出现购物车、订单、立即购买、支付等交易化文案。
- 宠物交易可以保留订单和交易行为，但不能把活体收养做成订单。
- 视觉仍遵守当前 M&D Stitch V2 规则：品牌为 M&D / meow & doggie，猫优先，doggie 友好，16dp 消费卡片，无装饰渐变块。
